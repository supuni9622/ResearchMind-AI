# Milestone 2.1 — Document Upload

**Status:** ✅ Completed

---

# Overview

Milestone **2.1 – Document Upload** establishes the foundation of the ResearchMind Knowledge Platform.

This milestone introduces the first complete vertical slice of the platform, allowing authenticated users to securely upload research documents through the API. Uploaded files are stored in Amazon S3 while document metadata is persisted in PostgreSQL.

This milestone intentionally focuses on reliable document ingestion rather than document understanding. AI processing such as parsing, chunking, embeddings, and retrieval will be implemented in subsequent milestones.

---

# Objectives

The objectives of this milestone were:

- Build a production-ready document upload workflow.
- Integrate AWS S3 as the primary document storage.
- Persist document metadata in PostgreSQL.
- Reuse AWS Cognito authentication.
- Implement validation before storage.
- Generate deterministic storage keys.
- Establish a clean service-oriented architecture.
- Create the foundation for the future RAG pipeline.

# Request Flow

1. User authenticates using AWS Cognito.
2. Client uploads a document through the upload endpoint.
3. JWT is validated.
4. Upload request is validated.
5. SHA-256 checksum is generated.
6. Deterministic storage key is generated.
7. File is uploaded to Amazon S3.
8. Metadata is persisted in PostgreSQL.
9. Upload response is returned.
10. Structured logs are emitted.

---

# Storage Strategy

ResearchMind separates binary content from metadata.

## Amazon S3

Stores:

- Original documents
- Future processed files
- Future OCR outputs
- Future extracted assets

Example:

```
documents/
    {owner_id}/
        {document_id}/
            original.pdf
```

---

## PostgreSQL

Stores metadata only.

Examples:

- Document ID
- Owner ID
- Filename
- Content Type
- Size
- Checksum
- Storage Key
- Status
- Timestamps

The database never stores document binaries.

# API

Current endpoint:

```
POST /api/v1/documents/upload
```

This endpoint accepts multipart form uploads and returns uploaded document metadata.

Future document APIs will follow the same REST resource.

```
GET    /documents
GET    /documents/{id}
DELETE /documents/{id}

POST   /documents/upload
POST   /documents/{id}/reprocess
POST   /documents/{id}/chunk
POST   /documents/{id}/embed
```
