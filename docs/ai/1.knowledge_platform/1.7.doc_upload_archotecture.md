# Architecture — Milestone 2.1 Document Upload

## Status

**Completed**

---

# Purpose

The Document Upload subsystem is the entry point into the ResearchMind Knowledge Platform.

Its responsibility is to securely receive research documents from authenticated users, validate them, store the original binary in Amazon S3, and persist document metadata in PostgreSQL.

This subsystem intentionally performs **document ingestion only**.

It does **not** parse, chunk, embed, or understand document content. Those responsibilities belong to later phases of the Knowledge Platform.

---

# Architectural Goals

This subsystem was designed with the following goals:

- Secure document uploads
- Separation of binary storage from metadata
- Modular service-oriented architecture
- Cloud-native storage
- Replaceable infrastructure components
- Clean separation of concerns
- Production-ready foundation for the RAG pipeline

# Component Responsibilities

## API Layer

Location

```
app/api/
```

Responsibilities

- Accept HTTP requests
- Authenticate user
- Validate request format
- Invoke UploadService
- Return HTTP response

The API layer contains **no business logic**.

---

## Upload Service

Location

```
app/ai/knowledge/upload/
```

Responsibilities

- Coordinate upload workflow
- Execute validation
- Generate checksum
- Generate storage key
- Upload file to storage
- Persist metadata
- Handle rollback
- Emit structured logs

The UploadService acts as the orchestration layer.

---

## Validation

Responsibilities

- Validate file type
- Validate file size
- Validate filename

Validation occurs before any storage operation.

---

## Hashing

Responsibilities

Generate SHA-256 checksum.

Current purpose

- Integrity verification

Future purpose

- Duplicate detection
- Incremental processing
- Cache lookup

---

## Storage Key Generator

Responsibilities

Generate deterministic S3 object keys.

Current format

```
documents/
    {owner_id}/
        {document_id}/
            original.pdf
```

Future files will reuse the same directory.

Example

```
documents/
    owner/
        document/
            original.pdf
            parsed.md
            metadata.json
            chunks.json
            embeddings.json
```

---

## Storage Layer

Current implementation

Amazon S3

Responsibilities

- Store original binary
- Delete object during rollback
- Abstract storage provider

The UploadService never communicates directly with boto3.

---

## Repository

Responsibilities

Persist metadata.

The repository never:

- uploads files
- validates files
- performs business logic

It only manages database persistence.

---

## PostgreSQL

Stores metadata only.

Current metadata includes

- Document ID
- Owner ID
- Filename
- Storage Key
- Content Type
- File Size
- Checksum
- Status
- Timestamps

The database never stores binary content.

# Failure Handling

The upload workflow is designed to maintain consistency between Amazon S3 and PostgreSQL.

Workflow

```
Upload File

↓

Persist Metadata

↓

Commit
```

If persistence fails after uploading the file, the uploaded object is deleted from Amazon S3.

This compensation step prevents orphaned files.

# Design Decisions

## Store binaries in Amazon S3

Reason

Object storage is the industry standard for document management and scales significantly better than storing binaries inside PostgreSQL.

---

## Store metadata in PostgreSQL

Reason

Metadata is relational and benefits from SQL querying, indexing, filtering, and future analytics.

---

## Deterministic storage keys

Reason

Stable storage paths simplify future processing pipelines and derivative asset generation.

---

## UploadService as orchestration layer

Reason

Business workflows should be coordinated by a dedicated service rather than controllers or repositories.

---

## Thin API layer

Reason

HTTP concerns remain isolated from business logic.

---

## Repository Pattern

Reason

Persistence logic remains isolated from business workflows.
