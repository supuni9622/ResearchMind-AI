# Implementation — Milestone 2.1 Document Upload

## Status

**Completed**

---

# Purpose

This document describes the implementation details of the Document Upload subsystem.

It serves as the primary developer reference for understanding how the upload workflow is implemented, how components interact, and how future enhancements should extend the subsystem.

---

# Objectives

The implementation delivers:

* Authenticated document uploads
* Secure document validation
* SHA-256 checksum generation
* Deterministic storage key generation
* Amazon S3 object storage
* PostgreSQL metadata persistence
* Structured logging
* Clean separation of concerns
* Production-ready service orchestration

# Upload Workflow

The upload workflow follows a layered architecture.

```
HTTP Request

↓

Authentication

↓

Upload Endpoint

↓

UploadService

↓

Validation

↓

Hashing

↓

Storage Key Generation

↓

Amazon S3 Upload

↓

Repository

↓

Database Commit

↓

Response
```

Each layer has a single responsibility.

---

# API Layer

Location

```
app/api/v1/documents.py
```

Responsibilities

* Receive multipart file uploads
* Authenticate the current user
* Calculate file size
* Delegate to UploadService
* Convert ORM entities into API schemas
* Return HTTP responses

The API layer contains no business logic.

---

# Authentication

Authentication is handled through AWS Cognito.

```
Request

↓

JWT

↓

get_current_user()

↓

ResearchMind User

↓

Upload Endpoint
```

The upload subsystem receives an authenticated `User` object and never validates JWTs directly.

---

# Upload Service

Location

```
app/ai/knowledge/upload/service.py
```

Responsibilities

* Coordinate the upload workflow
* Execute validation
* Generate checksum
* Generate storage key
* Upload the original document
* Persist metadata
* Commit transaction
* Roll back on failure
* Emit structured logs

The UploadService is the orchestration layer of this milestone.

---

# Validation

Location

```
app/ai/knowledge/upload/validators.py
```

Validation occurs before any storage operation.

Current validation rules include:

* Supported content type
* Maximum file size
* Filename validation

Future validation will include:

* Virus scanning
* Content inspection
* File signature verification

# Storage Key Generation

Location

```
app/infrastructure/storage/key_generator.py
```

Storage keys are deterministic.

Current format

```
documents/
    {owner_id}/
        {document_id}/
            original.pdf
```

This structure supports future derivative assets without changing the directory layout.

Examples

```
documents/
    owner/
        document/
            original.pdf
            parsed.md
            metadata.json
            chunks.json
            embeddings.json
```

# Error Handling

Upload failures trigger the following sequence.

```
Validation

↓

Hashing

↓

Storage

↓

Persistence
```

If metadata persistence fails after the document has already been uploaded, the uploaded object is removed from Amazon S3 to prevent orphaned files.

The database transaction is rolled back.

---

# Logging

Structured logs are emitted for:

* Successful uploads
* Upload failures
* Storage cleanup failures

Each upload records:

* Document ID
* Owner ID
* Filename
* Storage Key
* File Size
* Content Type
* Checksum
* Upload Duration

# Implementation Decisions

The following implementation decisions are frozen.

## Amazon S3

All original documents are stored in Amazon S3.

---

## PostgreSQL

Only metadata is persisted in PostgreSQL.

---

## Thin Controllers

HTTP endpoints delegate immediately to UploadService.

---

## Service Orchestration

Business workflows are coordinated by UploadService.

---

## Repository Pattern

Repositories own persistence only.

---

## Infrastructure Isolation

Infrastructure components remain behind abstractions.

---

## Deterministic Storage Keys

Storage paths are predictable and stable.

---

# Current Limitations

The upload subsystem intentionally excludes:

* OCR
* Parsing
* Metadata extraction
* Duplicate detection
* Async processing
* Chunking
* Embeddings
* Vector indexing
* Retrieval

These capabilities will be implemented in later milestones.
