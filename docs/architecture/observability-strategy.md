# Observability Strategy — Logging

## Scope

This document currently covers **logging only**, which is the only
observability pillar actually implemented. Metrics (Prometheus/Grafana),
distributed tracing (OpenTelemetry), and LLM tracing (LangSmith) have
placeholder docs under `docs/monitoring/` but no implementation yet.

---

# Framework

Logging is `structlog`, configured once in `app/core/logging.py` via
`configure_logging()`, called from `lifespan()` in `app/core/lifespan.py` at
application startup — before anything else runs.

Every module gets a logger the same way:

```python
import structlog
logger = structlog.get_logger()
```

Application services that want per-call context bind a scoped logger
instead of using the module-level one, e.g. in `ProcessingService`:

```python
log = logger.bind(document_id=str(request.document_id), owner_id=owner_id, ...)
```

All subsequent `log.info(...)` / `log.debug(...)` calls in that function
automatically carry those fields without repeating them.

# Renderer Strategy

- **Development** (`settings.environment != "production"`): coloured,
  human-readable output via `structlog.dev.ConsoleRenderer()`.
- **Production**: `structlog.processors.ExceptionRenderer()`.

> **Known issue**: the docstring in `configure_logging()` says production
> uses `JSONRenderer` (machine-parseable, one line per event), but the
> implementation actually wires up `ExceptionRenderer`, which formats
> exception tracebacks — it is not a general-purpose structured renderer.
> This looks like an unintentional mismatch between the comment and the
> code and should be verified/fixed before relying on JSON log output in
> production (e.g. for log aggregation in CloudWatch/Datadog).

Stdlib loggers (uvicorn, SQLAlchemy, etc.) are bridged through the same
processor chain via `ProcessorFormatter`, so third-party log lines share
the same format and the same contextvars (e.g. `request_id`) as
application logs.

Noisy stdlib loggers are capped at `WARNING`:

- `uvicorn.access` (replaced by `LoggingMiddleware`)
- `sqlalchemy.engine`
- `httpx`
- `httpcore`

# Log Levels In Practice

The root logger and structlog's filtering wrapper are both set to `INFO`.

```python
wrapper_class=structlog.make_filtering_bound_logger(logging.INFO)
root.setLevel(logging.INFO)
```

**This means every `log.debug(...)` / `logger.debug(...)` call in the
codebase is currently suppressed by default.** There are a meaningful
number of them (parser resolution, artifact-write steps, S3 debug traces,
hashing completion) intended for local troubleshooting — to see them, the
level needs to be lowered (e.g. via an env-driven override; there isn't one
today).

Level usage convention across the codebase:

| Level | Used for |
|---|---|
| `debug` | Fine-grained pipeline steps not needed in normal operation (parser resolved, artifact written, token verified, DB session events) |
| `info` | Lifecycle milestones and successful outcomes (`document.uploaded`, `processing.completed`, `user.created`, `app.ready`) |
| `warning` | Expected/recoverable failures (validation rejections, auth token issues, health-check failures, storage cleanup failures) |
| `exception` | Unhandled or pipeline-terminating failures — captures the traceback automatically |

# Correlation

`LoggingMiddleware` (`app/middleware/request_logging.py`) binds
per-request context via `structlog.contextvars` so every log line emitted
during that request — across every module — carries the same fields
without threading them through function signatures:

```python
structlog.contextvars.bind_contextvars(
    request_id=request_id,
    method=request.method,
    path=request.url.path,
    client=request.client.host if request.client else None,
)
```

`request_id` is generated by `RequestIDMiddleware` (or reused if already
present on `request.state`), echoed back to the client as the
`X-Request-ID` response header, and cleared from contextvars at the end of
each request.

Once a request authenticates, `get_current_user`
(`app/auth/dependencies.py`) additionally binds `user_id`, so every log
line for the rest of that request is attributable to a specific user.

Middleware order (`app/middleware/register.py`): CORS → Request ID →
Request Logging → Request Timing → route handler.

# Event Catalog

Event names are dot-namespaced by module/action, e.g. `<domain>.<event>`.
This is the full set of events currently emitted, grouped by area.

## HTTP / lifecycle

`http.request`, `http.response`, `app.starting`, `app.ready`,
`app.shutting_down`, `app.shutdown_complete`, `app.unhandled_exception`,
`db.migrations_starting`, `db.migrations_complete`

## Health

`health.postgres_unhealthy`, `health.valkey_unhealthy`,
`health.qdrant_unhealthy`

## Authentication

`auth.token_expired`, `auth.token_invalid_audience`,
`auth.token_invalid_issuer`, `auth.token_invalid`,
`auth.token_wrong_type`, `auth.token_verified`,
`auth.code_exchange_started`, `auth.code_exchange_succeeded`,
`auth.code_exchange_failed`

## Users

`user.created`, `user.not_found`, `user.synced`, `user.first_login`,
`user.last_login_updated`, `user.deactivated`

## Upload

`upload.validation_failed` (with `reason`: `empty_filename`,
`unsupported_extension`, `unsupported_content_type`, `empty_file`,
`file_too_large`), `document.uploaded`, `document.duplicate_detected`
(carries `existing_document_id`; logged just before `UploadService.upload()`
raises `ConflictException` — a duplicate is not routed through
`document.upload_failed`, since nothing was written and there is nothing to
roll back or clean up), `document.upload_failed` (carries `stage`:
`hashing` / `storage_upload` / `persistence`),
`upload.storage_cleanup_succeeded`, `upload.storage_cleanup_failed`

## Processing

`processing.started`, `processing.parser_resolved`,
`processing.document_downloaded`, `processing.parse_completed`,
`processing.parse_failed`, `processing.artifacts_built`,
`processing.artifact_persistence_failed`, `processing.completed`,
`registry.parser_registered`, `registry.parser_replaced`,
`registry.parser_not_found`, `artifact_writer.write.started`,
`artifact_writer.write.completed`, `artifact_writer.artifact.write_failed`,
`artifact_writer.artifact.written`

## Document lifecycle orchestration

`document_processing.started`, `document_processing.status_updated`,
`document_processing.status_update_failed`,
`document_processing.completed`, `document_processing.failed`,
`document_processing.failed_status_persist_failed`

## Infrastructure

`hasher.sha256_complete`, `hasher.sha256_failed`, `s3.client_initialised`,
`s3.upload_complete`, `s3.upload_failed`, `s3.download_complete`,
`s3.download_failed`, `s3.delete_complete`, `s3.delete_failed`,
`s3.exists_check_failed`, `s3.presigned_url_generated`

## Global error handling

`app.exception` (any `AppException` subclass), `app.validation_error`
(FastAPI request validation failures)

---

# Where Logs Go

Everything writes to stdout (`logging.StreamHandler(sys.stdout)`). There is
no log shipping, aggregation, or retention configured yet — that's tracked
as future work under `docs/monitoring/`.
