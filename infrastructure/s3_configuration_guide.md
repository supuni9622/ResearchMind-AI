# S3 Infrastructure Configuration Guide

**Project:** ResearchMind AI
**Environment:** Development
**Version:** 1.0
**Last Updated:** June 2026

---

# Overview

Before implementing document uploads in the application, we must provision and verify our AWS S3 infrastructure.

This document describes the recommended S3 configuration for the ResearchMind project, following AWS best practices and the principle of least privilege.

> **Goal:** Ensure the storage infrastructure is fully configured and verified before writing any application code.

---

```text
                 FastAPI Backend

                       │ boto3

               Amazon S3 Bucket

             Private Storage Only
                       │
          IAM User / IAM Role Access
                       │
```

# Step 0 — Create the S3 Bucket

Create a dedicated bucket for the development environment.

**Recommended bucket name**

```text
researchmind-ai-documents-dev
```

Use environment-specific buckets because the project will eventually have multiple deployment environments.

Future bucket structure:

```text
researchmind-ai-documents-dev
researchmind-ai-documents-staging
researchmind-ai-documents-prod
```

Keeping environments isolated prevents accidental data sharing and simplifies deployment.

---

# Step 1 — Select the AWS Region

Use the same AWS Region as the Cognito User Pool.

**Recommended region**

```text
ap-southeast-1
```

Using a single region provides:

* Lower latency
* Simpler IAM configuration
* Easier infrastructure management
* Lower cross-region data transfer costs

---

# Step 2 — Configure the S3 Bucket

## Object Ownership

Configure:

```text
Bucket owner enforced
```

This disables ACLs and ensures all uploaded objects are owned by the bucket owner.

---

## Block Public Access

Enable **all four** public access settings.

```text
✅ Block all public access
```

The bucket should remain completely private.

Users will access files through pre-signed URLs generated by the backend.

---

## Bucket Versioning

Enable versioning.

```text
Status:
Enabled
```

Benefits:

* Document history
* Recovery from accidental deletion
* Protection against overwrites
* Future audit capabilities

---

## Default Encryption

Enable server-side encryption.

```text
SSE-S3
```

This uses AWS-managed encryption keys.

KMS can be introduced later if customer-managed encryption is required.

---

## Bucket Tags

Add the following tags.

| Key         | Value        |
| ----------- | ------------ |
| Project     | ResearchMind |
| Environment | dev          |

These tags simplify billing, monitoring, and resource management.

---

# Step 3 — Define the Object Key Structure

Amazon S3 is a flat object store.

Folders are logical prefixes rather than physical directories.

Use the following naming convention.

```text
documents/
```

Per user:

```text
documents/{user_id}/
```

Per document:

```text
documents/{user_id}/{document_id}/
```

Original uploaded file:

```text
documents/{user_id}/{document_id}/original.pdf
```

Future project prefixes:

```text
documents/
processed/
reports/
exports/
datasets/
benchmarks/
```

These prefixes do **not** need to be created manually.

Amazon S3 automatically creates them when objects are uploaded.

# Step 4 — Configure IAM Permissions

Never use the AWS root account for application access.

Create a dedicated IAM user for development.

Example:

```text
researchmind-api-dev
```

---

## Create a Custom IAM Policy

Grant access only to the ResearchMind bucket.

Example policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ResearchMindObjectAccess",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject"
      ],
      "Resource": [
        "arn:aws:s3:::researchmind-ai-documents-dev/*"
      ]
    },
    {
      "Sid": "ResearchMindBucketAccess",
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::researchmind-ai-documents-dev"
      ]
    }
  ]
}
```

> **Note**
>
> Although the S3 API provides a `HeadObject` operation, there is **no** corresponding IAM permission named `s3:HeadObject`.
>
> `HeadObject` requests are authorized using:
>
> ```text
> s3:GetObject
> ```

---

## Why We Don't Use AmazonS3FullAccess

Avoid attaching the AWS-managed **AmazonS3FullAccess** policy.

It grants unnecessary permissions such as:

* Listing every bucket
* Creating buckets
* Deleting buckets
* Modifying bucket policies
* Accessing unrelated project data

ResearchMind follows the **Principle of Least Privilege**, granting only the permissions required by the application.

# Step 5 — Configure Local AWS Credentials

Install and configure the AWS CLI.

Run:

```bash
aws configure
```

Provide:

```text
AWS Access Key ID
AWS Secret Access Key
Default Region
Output Format (json)
```

Example:

```text
Default Region:
ap-southeast-1

Default Output Format:
json
```

AWS CLI stores credentials in:

```text
~/.aws/credentials
```

Configuration is stored in:

```text
~/.aws/config
```

---

## Verify the IAM Identity

Run:

```bash
aws sts get-caller-identity
```

Example output:

```json
{
    "Account": "123456789012",
    "Arn": "arn:aws:iam::123456789012:user/researchmind-api-dev"
}
```

This confirms the application is using the intended IAM user.

---

# Step 6 — Verify Bucket Access

Before writing Python code, verify the infrastructure from the command line.

---

## List Objects

```bash
aws s3 ls s3://researchmind-ai-documents-dev
```

If the bucket is empty, the command will simply return no objects.

---

## Upload a Test File

```bash
echo "ResearchMind" > test.txt
```

Upload it.

```bash
aws s3 cp test.txt s3://researchmind-ai-documents-dev/
```

---

## Verify Upload

```bash
aws s3 ls s3://researchmind-ai-documents-dev
```

You should see:

```text
test.txt
```

---

## Delete the Test File

```bash
aws s3 rm s3://researchmind-ai-documents-dev/test.txt
```

---

## Successful Verification

If all commands succeed, the following components are correctly configured:

* AWS credentials
* IAM policy
* S3 bucket permissions
* Network connectivity
* Region configuration

At this point the infrastructure is ready for application development.

---

# Step 7 — Configure Environment Variables

Update the project's `.env` file.

```env
AWS_REGION=ap-southeast-1
AWS_S3_BUCKET=researchmind-ai-documents-dev
```

Do **not** store AWS access keys in `.env` for local development.

---

# AWS Credential Provider Chain

Boto3 automatically resolves credentials using the AWS Credential Provider Chain.

Order of precedence:

1. Environment variables
2. Shared credentials file (`~/.aws/credentials`)
3. AWS configuration (`~/.aws/config`)
4. IAM Roles (EC2, ECS, Lambda)

This is the recommended approach because:

* Secrets are not committed to source control.
* Developers maintain separate credentials.
* Production environments use IAM roles without code changes.

---

# Development Best Practices

* Never use the root account for applications.
* Grant only the minimum IAM permissions required.
* Keep the bucket private.
* Access files using pre-signed URLs.
* Enable bucket versioning.
* Enable server-side encryption.
* Use environment-specific buckets.
* Keep all AI infrastructure within the same AWS Region.
* Use the AWS Credential Provider Chain instead of embedding credentials in code or `.env`.
